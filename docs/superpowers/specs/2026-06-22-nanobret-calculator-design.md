# NanoBRET™ PPI Calculator — Design Spec

## Purpose

A new Hub app implementing the calculation pipeline from Promega's NanoBRET™ Protein:Protein Interaction (PPI) Assay protocol (TM439, Rev 10/23). Users paste/import raw donor (460nm) and acceptor (618nm) plate-reader values; the app computes BRET ratios, mBU, corrected mBU, assay-quality metrics (Z′/Z factor), and — for dose-response experiments — fits a 4-parameter logistic curve to extract IC50/EC50.

Out of scope for this app: the wet-bench protocol itself (transfection, cell seeding, ligand titration), the Donor Saturation Assay (DSA) mode (Section 10.B of the protocol — a separate specificity-validation workflow with a different plot shape), kinetic time-course readouts, and multi-compound/multi-curve comparison in a single analysis (each analysis run covers one compound/curve, matching the protocol's per-compound dose series).

## Architecture

Single self-contained HTML file, `NanoBRET/nanobret.html`, following every existing app's pattern (inline `<style>` + `<script>`, no build step, no external runtime dependencies beyond what's already vendored elsewhere in the suite). Integrated into the Hub shell the same way every other app is: a home-screen card, an `APP_INFO` entry, a base64-embedded payload generated by `embed.py`, and an iframe-hosted view — following the exact steps documented in `CLAUDE.md`'s "How to add a new app" section.

Four tabs, matching the suite's established tab-bar convention (Echo, Spectra, Degradation Explorer, etc. — icon + label, active-state underline, identical CSS skeleton reused verbatim and re-themed to the app's own accent):

1. **Plate Data** — raw donor/acceptor value entry for both channels.
2. **Endpoint / QC** — named experimental conditions (well ranges), corrected mBU, Z′/Z factor.
3. **Dose-Response** — compound dose series, shared no-ligand baseline, 4PL curve fit, IC50/EC50.
4. **Guide** — static reference tab, matching every other app's Guide tab pattern (icon-headed sections, brief explanatory text, no interactivity), now also carrying the line-icon-per-section treatment shipped suite-wide in Tier 3 (v1.2.2).

Data flows one direction, top to bottom: Plate Data tab holds the two raw value grids (donor, acceptor) in app state → Endpoint/QC tab reads from that state via user-defined well-range groups, computing raw ratio → mBU → corrected mBU → Z′/Z per group → Dose-Response tab reads the same raw state via a different grouping (dose series + one shared baseline), computing corrected mBU per dose point and feeding that into the 4PL fitter. No tab mutates another tab's input; all derived values are computed live from the single raw-data source, the same one-way-derivation pattern Spectra and Echo already use.

## Plate Data tab

- A **format toggle** at the top of the tab: "96-well" / "384-well" (Echo and Spectra already support both; reuse their plate-dimension constants and grid-rendering approach rather than redefining them).
- **Two CSV import slots**, one for the donor (460nm) channel and one for the acceptor (618nm) channel, each accepting a pasted or file-loaded plate-shaped CSV (rows = letters, columns = numbers) — this directly reuses Spectra's existing grid-CSV parser (the same one backing its Plate Reader tab), pointed at two separate grids instead of one.
- A **combined preview**: once both channels are loaded, render a single plate grid where hovering/selecting a well shows both raw values (donor and acceptor) for that well side by side, plus the computed raw ratio. This is new UI (no existing app shows two values overlaid per well) but is a thin display layer on top of the existing plate-grid renderer, not a new grid engine.
- Manual entry is also supported per-well (clicking a well opens the same small inline editor Spectra uses for manual corrections), for users who only have a few wells to fix rather than re-importing a whole CSV.

## Endpoint / QC tab

- Users define **named conditions**, each backed by a **well-range text input** (e.g. `A1-A3, B1-B3`) — this reuses Echo's well-range parser verbatim (the same syntax, comma-separated ranges/singles, already validated and tested there).
- Two condition roles, matching the protocol's Section 5.F vocabulary exactly: **"+Ligand"** conditions and **"No-Ligand"** (background) conditions. A user can define multiple named conditions of each role (e.g. several test constructs, each with its own +Ligand wells, sharing or each defining their own No-Ligand wells).
- Per condition, computed and displayed:
  - **Raw Ratio** = Acceptor(618nm) / Donor(460nm), per well.
  - **mBU** = Raw Ratio × 1000, per well.
  - **Mean mBU** and **SD** across the condition's wells.
  - **Corrected mBU** = Mean mBU(+Ligand) − Mean mBU(No-Ligand), paired per matching construct.
- A **Z′ factor** card, computed per +Ligand/No-Ligand pair using the protocol's exact formula:

  ```
  Z′ = 1 − [3×SD(+Ligand) + 3×SD(No-Ligand)] / |Mean(+Ligand) − Mean(No-Ligand)|
  ```

- An optional **Z factor** card (treated vs. untreated framing, same formula shape, swapping in user-labeled "treated"/"untreated" condition pairs instead of +Ligand/No-Ligand) for users running compound-treatment QC rather than a pure ligand-titration QC.
- Z′/Z values get the conventional traffic-light read-out already used informally in assay QC tooling: ≥0.5 good (green), 0–0.5 marginal (amber), <0 unreliable (red) — purely a presentational threshold, not part of the Promega formula itself.

## Dose-Response tab

- A **single compound/curve per analysis**, matching the protocol's Section 5.C framing: one dose-response experiment = one compound's titration against one shared baseline.
- **Dose series rows**: each row is a concentration value + a well-range (reusing the same well-range parser as the Endpoint/QC tab). All dose-series wells contain ligand; only the compound concentration varies between rows.
- **One shared No-Ligand baseline** well-range, used for background subtraction across every dose point (per protocol Section 5.C — a single baseline, not one per dose).
- Per dose point: corrected mBU = Mean mBU(dose well group) − Mean mBU(shared baseline).
- **4PL curve fit** over (concentration, corrected mBU) pairs, reusing Echo's existing 4-parameter logistic fitter (already implemented and tested there for its own dose-response curves) rather than writing a new fitting routine.
- Output: fitted curve overlaid on the scatter of corrected mBU points (log-concentration x-axis, matching Echo's existing dose-response chart styling), plus the fitted **IC50/EC50** value with its fit-confidence indicator (reusing whatever confidence/R² indicator Echo already surfaces for its own fits).

## Guide tab

Static reference content, structured the same way every other app's Guide tab is post-Tier-3: icon-headed sections with a short explanation each. Sections: what the Plate Data tab expects, how Endpoint/QC's well-range and condition-role inputs work, how to read Z′/Z, how the Dose-Response tab's baseline/dose-series split works, and a one-paragraph plain-language explanation of what NanoBRET/BRET ratio physically measures (donor NanoLuc emission at ~460nm vs. acceptor HaloTag-ligand emission at ~618nm) for users unfamiliar with the assay.

## App identity

- **App ID:** `nanobret`
- **Display name:** NanoBRET Calculator
- **Accent color:** `#5e72c4` (muted indigo-blue), distinct from every existing app's accent, nodding to NanoLuc's blue bioluminescent donor emission.
- **Tab icons:** reusing established cross-suite glyphs where the concept matches —
  - Plate Data → the 9-dot well-grid icon (already used by Spectra/Echo's plate-related tabs).
  - Dose-Response → the dose-response curve glyph (already used by Echo's own dose-response tab).
  - Guide → the open-book glyph (suite-wide Guide tab convention).
  - Endpoint/QC → **new glyph**: a small gauge/checkmark-circle, read as "quality check" (no existing suite icon fits this concept).
- **Hub home card:** added to `hub-shell.html` following the established pattern (bordered box, accent-color background, white-stroke inline SVG pictogram). Icon concept: a simplified BRET donor→acceptor energy-transfer glyph (two overlapping circles with a small arrow/wave between them), distinct from every other home card's glyph.

## Testing

Manual verification via Playwright, matching the pattern used for every other app in this suite's recent rounds: load the app in the Hub iframe, exercise CSV import on both Plate Data channels, define Endpoint/QC conditions and confirm Z′/Z values match hand-calculated expected values for a known synthetic dataset, run a dose-response series and confirm the fitted IC50 lands within a reasonable tolerance of a synthetic dataset's known true IC50, screenshot all four tabs in both light and dark theme at desktop and mobile widths, confirm zero console errors.

Numerical correctness of the core formulas (raw ratio, mBU, corrected mBU, Z′/Z, 4PL fit) should additionally be checked against hand-computed values from a small fixed synthetic dataset embedded directly in the implementation plan's tasks — these are pure functions with no DOM dependency, so they're straightforward to unit-test in isolation before wiring up the UI.
